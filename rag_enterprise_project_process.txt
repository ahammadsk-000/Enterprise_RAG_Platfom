0. WHAT THIS PROJECT IS
--------------------------------------------------------------------------------
A production-style, multi-tenant Retrieval-Augmented Generation (RAG) platform for
enterprise knowledge — similar in spirit to AnythingLLM / Dify / enterprise ChatGPT.
Users upload documents; the system parses, chunks, embeds and indexes them, then
answers questions with citations grounded in those documents. It also provides a
knowledge graph, multi-agent research, RAG evaluation, observability, and a full
web UI.

Built FastAPI-first with clean architecture + Domain-Driven Design, async
everywhere, provider abstractions for all AI/infra so components are swappable, and
a React UI. Ships in two run modes: a lightweight "demo mode" (laptop-friendly) and
a full Docker stack (real models + persistent infra).

1. HIGH-LEVEL ARCHITECTURE
--------------------------------------------------------------------------------
Clients (React SPA / REST / WebSocket)
        |  HTTPS / WSS
   Nginx (edge: TLS, rate-limit) [prod]
        |
   FastAPI API gateway  --- AuthN/Z, RBAC, tenant scoping, dependency injection
        |          \            \              \
   Domain services   RAG engine   Agents (graph)  Ingestion (enqueue)
        |              |             |               |
   PostgreSQL       Retrieval     LangGraph-style   Celery workers
   (system of       (dense+BM25+  state machine     (ocr/parse/chunk/
    record)          rerank+RRF)                     embed/index/graph)
        |              |   \                              |
   Qdrant (vectors)  Neo4j (graph)   Redis (cache/broker)  Object store (S3/MinIO)
        |
   Observability: OpenTelemetry -> collector; Prometheus /metrics; Grafana; Loki

Design rules:
- Clean Architecture + DDD: each business "domain" owns models, schemas,
  repositories, services. Imports point inward (api -> services -> repo interfaces).
- Async-first: all request + DB paths are async; heavy work goes to Celery workers.
- Event-driven ingestion pipeline: upload -> parse/OCR -> chunk -> embed -> index
  -> graph-extract, each a stage with its own job record.
- Provider abstractions (Protocols) for LLM, embeddings, vector store, graph store,
  OCR, object storage, reranker — selected by a factory from settings; fakes/
  in-memory variants used in tests and demo mode.
- Multi-tenancy: every tenant-owned row carries organization_id; queries are
  tenant-scoped; vector/graph payloads filtered by tenant + workspace.

4. IMPLEMENTATION PHASES (what was built, in order)
--------------------------------------------------------------------------------
Each phase was implemented, tested, and committed separately on the `main` branch.

PHASE 1 — Foundation + Identity/RBAC/SSO
  - Architecture docs; monorepo scaffold; backend core (Pydantic settings, structlog,
    RFC9457 errors, request-context middleware, async DB session/Base, app factory,
    health probes); docker-compose; Alembic; Celery app.
  - Identity domain: organizations, users, roles, permissions, memberships, api_keys,
    oauth_accounts. argon2 password hashing; JWT access/refresh with rotation.
  - RBAC: 18 permission codes, 4 system roles (owner/admin/member/viewer);
    require_permission dependency + tenant guard.
  - Auth routes (register/login/refresh/me); OAuth2/OIDC SSO (Google/Microsoft via
    Authlib, config-gated); users/members. Migration 0001 seeds permissions/roles.

PHASE 2 — Document ingestion + OCR
  - documents + ingestion_jobs models; status/stage enums; migration 0002.
  - ObjectStorage abstraction (S3/MinIO + in-memory + factory).
  - OCR engine abstraction (Tesseract + null + factory).
  - Parser registry: plain-text, HTML (stdlib), PDF (PyMuPDF + OCR fallback), DOCX,
    image; MIME resolution + validation.
  - Metadata: SHA-256 content hashing (dedupe), language detection, counts.
  - DocumentService (upload+dedupe/list/get/status/delete) + IngestionService
    orchestrator (PARSE -> METADATA, per-stage job rows). Celery ingest task +
    TaskBus abstraction. Document REST API.

PHASE 3 — Chunking + Embeddings + Vector DB
  - chunks + embedding_versions models; migration 0003.
  - Chunkers (registry, per-workspace strategy): recursive (default), parent/child,
    semantic (sentence-aware), table-aware, code-aware; token estimation.
  - EmbeddingProvider Protocol (Ollama/SentenceTransformers/OpenAI/fake) + factory;
    embedding cache keyed by version:content_hash.
  - VectorStore Protocol (Qdrant + in-memory) + factory; one collection per org.
  - ChunkingService (idempotent) + EmbeddingService (batched, cached, upserts
    vectors, records vector_id + embedding_version_id).
  - Ingestion extended: PARSE -> METADATA -> CHUNK -> INDEX -> INDEXED.
  - Document reindex endpoint; chunk_count in status; vector cleanup on delete.

PHASE 4 — Hybrid retrieval + RAG engine + citations
  - LLM provider abstraction (Ollama/OpenAI/fake) with generate + stream;
    cross-encoder reranker (+ null) behind factories.
  - DenseRetriever (vector ANN, hydrates content from Postgres) + BM25Retriever
    (PostgreSQL full-text search; GIN expression index from migration 0004).
  - Reciprocal Rank Fusion (RRF) for hybrid search; context compression
    (dedupe + token budget); query expansion (identity default, optional LLM).
  - RetrievalService: strategy-aware (dense/bm25/hybrid) -> fuse -> rerank -> top_k,
    with per-query retrieval_logs.
  - RAG engine: citation-grounded prompt; citation marker parsing/resolution;
    confidence scoring (retrieval strength + grounding); RagService.
  - API: POST /search (retrieval only), POST /rag/query (grounded answer + citations).

PHASE 5 — Streaming chat + workspaces + memory
  - Workspace model + CRUD; migration 0005 also backfills workspace_id FKs on
    documents/chunks/memberships. Long-term Memory model + repo.
  - Conversation + Message models (chat history = conversation memory).
  - ChatService: conversation CRUD + grounded answers reusing the retrieval/citation
    engine, with recent history injected into the prompt; non-streaming answer() and
    streaming stream_answer() async generator.
  - WebSocket endpoint /api/v1/ws/chat/{conversation_id} (token auth; streams token
    events then a done event with citations/confidence). REST chat routes.

PHASE 6 — Graph RAG (knowledge graph)
  - GraphStore abstraction: Neo4j (lazy async driver) + in-memory + factory;
    per-tenant isolation, document_id provenance on relations.
  - Entity/relation extractors: rule-based (deterministic) and LLM-based (JSON output
    with rule-based fallback) + factory.
  - GraphBuilder (extract -> upsert) and GraphRetrievalService (query -> seed
    entities -> k-hop neighbor traversal). Ingestion gains optional GRAPH stage.
  - POST /graph/explore endpoint.

PHASE 7 — Multi-agent retrieval
  - agent_runs + agent_steps models; migration 0006.
  - ResearchAgentGraph: typed state machine planner -> retriever -> summarizer ->
    verifier -> citation, with a single re-retrieval loop on failed verification.
    Dependency-free orchestrator shaped like a LangGraph StateGraph (swappable).
  - AgentService persists run + per-node steps; POST /agents/research.

PHASE 8 — RAG evaluation
  - Heuristic metrics (deterministic, dependency-free, CI-friendly): faithfulness,
    answer_relevancy, context_precision, context_recall, hallucination — pluggable
    behind evaluate_sample for an LLM/RAGAS upgrade.
  - eval_datasets/eval_samples/eval_runs/eval_results models; migration 0007.
  - EvalService: create golden datasets, run offline eval (answer from gold contexts
    -> score -> persist per-sample + aggregate). Endpoints: datasets create/list, run.

PHASE 9 — Observability
  - Prometheus metrics (http_requests_total + http_request_duration_seconds) recorded
    in middleware; exposed at /metrics; prometheus_client optional (no-op if absent).
  - OpenTelemetry: best-effort FastAPI instrumentation + OTLP export, gated by
    settings/availability (never breaks startup).
  - Admin analytics: AnalyticsService aggregates tenant usage (documents,
    conversations, messages, tokens, avg confidence, retrieval queries + latency,
    agent runs); GET /admin/analytics/overview.

PHASE 10 — Kubernetes/Helm + CI + hardening
  - Rate limiting (in-memory fixed-window, per-client) + security-headers middleware.
  - Helm chart: API Deployment+Service+HPA, per-queue worker Deployments, beat,
    Ingress (TLS/cert-manager), ConfigMap+Secret+ServiceAccount, pre-upgrade Alembic
    migration Job (helm hook); values + values-staging.
  - GitHub Actions CI: backend (ruff/mypy/pytest unit + Postgres integration),
    frontend build, docker build + Trivy scan on main.

FRONTEND (full platform UI)
  - React SPA wired to the backend (Vite dev proxy includes ws:true for chat socket).
  - Pages: Login/Register; Chat (WebSocket streaming, conversation sidebar, citations,
    confidence; auto-titled conversations; rename/delete; "no relevant sources"
    handling); Search (hybrid/dense/bm25 + rerank toggle); Documents (upload, status
    polling, reindex, delete, in-app editor, download, extract-to-markdown); Graph
    explorer; Agents (answer + plan + step timeline); Evaluation runner; Workspaces
    (CRUD + sidebar switcher); Admin analytics dashboard.

DOCUMENT EDITOR (added after core phases)
  - In-app CodeMirror 6 editor for text files (txt/md/html/csv/json/code):
    native spellcheck (browser), dictionary + document-word autocomplete, and
    AI inline completion ("ghost text", Tab to accept) via POST /assist/complete.
  - Save -> stores new bytes, drops stale vectors, re-runs ingestion (re-index).
  - Download (raw file or edited buffer).
  - "Extract -> MD" for binary docs (PDF/DOCX): creates an editable .md from the
    already-extracted text (indexed); original kept and downloadable.

9. HOW TO RUN
--------------------------------------------------------------------------------
A) DEMO MODE (recommended for laptops, ~1 GB) — enterprise_rag_demo.bat
   - Needs: Docker Desktop running (for ONE small Postgres container), Python 3.12,
     Node 20+.
   - The launcher: starts a Postgres container; creates backend/.venv and installs
     requirements-demo.txt; runs `alembic upgrade head`; starts native uvicorn
     (ENVIRONMENT=test, INGESTION_INLINE=true) + native Vite frontend; opens
     http://localhost:5173.
   - Uses fake LLM/embeddings + in-memory vector/graph/object stores + inline
     ingestion. Retrieval, chunking, citations, graph, agents, eval are all REAL;
     only the LLM/embeddings are stubbed and the in-memory stores reset on restart.
   - After a backend restart, re-upload documents (in-memory stores are cleared;
     Postgres rows persist).

B) FULL STACK (~10 GB, real models) — enterprise_rag.bat
   - Needs: Docker Desktop; ~10 GB RAM; one-time Ollama model pull:
        docker compose exec ollama ollama pull llama3.1:8b
        docker compose exec ollama ollama pull nomic-embed-text
   - The launcher: brings up the full docker-compose stack (api, worker, beat,
     postgres, redis, qdrant, neo4j, minio, ollama, observability, frontend),
     runs migrations, opens the app. Real LLM answers, real semantic search,
     persistent Qdrant/Neo4j/MinIO, Celery worker, Tesseract OCR, Prometheus/Grafana.

C) MANUAL (developers)
   - Backend: cd backend ; pip install -e ".[dev]" ; alembic upgrade head ;
     uvicorn app.main:app --reload   (needs Postgres; configure backend/.env)
   - Frontend: cd frontend ; npm install ; npm run dev

DEMO vs FULL: same UI, same features, same code paths. Difference is AI quality +
persistence: demo = fake LLM/embeddings + in-memory stores; full = real Ollama
models + Qdrant/Neo4j/MinIO that persist across restarts.

11. KEY DESIGN DECISIONS & NOTABLE FIXES (history that explains the code)
--------------------------------------------------------------------------------
- Provider Protocols + factories everywhere -> demo and tests use fakes/in-memory
  with zero changes to business logic; heavy libs are imported lazily so the app
  runs without them.
- Inline ingestion (InlineTaskBus) for demo: runs the real pipeline in-process via
  asyncio so no Celery/Redis/worker is needed on a laptop.
- BM25 = PostgreSQL full-text search (to_tsvector @@ plainto_tsquery) with a GIN
  expression index — no extra search engine; hybrid fuses it with dense via RRF.
- Demo launcher runs uvicorn WITHOUT --reload so editing files mid-session doesn't
  restart the backend and wipe the in-memory stores.
- Chat: fixed a race where creating a conversation triggered a history-load that
  overwrote the streaming answer; history now loads only on explicit selection.
- Conversations auto-title from the first question; support rename + delete.
- When retrieval returns no context, Chat/RAG short-circuit with a clear
  "no relevant sources" message instead of a hallucinated answer.
- Graph in-memory store de-duplicates neighbors so the explorer shows distinct
  entities.
- Documents: duplicate uploads are detected by content hash and surfaced in the UI;
  PDF/DOCX support added to demo via pymupdf + python-docx (scanned PDFs still need
  the full-stack OCR).
